# MVPメリデメ比較：「ブラウザによらずアクセス不可」の実現方式（2026-06-25）

## 前提の整理

未確定事項は「サイトブロックをブラウザに依存させない範囲をどこまでMVPでやるか」。
ただし**アプリ自体のブロック（例：YouTubeアプリ）はこの論点とは別**で、どちらの案でも
「在宅判定中、対象アプリがフォアグラウンドに来たら即座にホームへ戻す/オーバーレイで遮断する」
という単純な仕組み（`UsageStatsManager`等でフォアグラウンドアプリを監視）で対応可能。
両案の差が出るのは**「指定サイトを、どのブラウザで開いても止める」**という部分のみ。

## 案A：VPN/DNSベースで最初から全ブラウザ・全アプリ共通ブロック

`VpnService`でローカルVPN（ループバックのnull routing）を構築し、デバイス上の全通信を経由させて
DNSクエリを監視。指定ドメイン（例：`*.youtube.com`、`*.googlevideo.com`）への問い合わせを
NXDOMAIN/sinkholeで潰す。root不要、ブラウザ・アプリの種類を問わず同じ仕組みで効く。

| 軸 | 評価 |
|---|---|
| ブラウザ非依存の堅牢性 | ◎ 仕組み上ブラウザを一切見ないので、どのブラウザ・WebViewでも効く |
| 実装コスト | △〜× VpnService実装、パケット/DNSパース、sinkhole応答生成など独自に組む必要がある（参考になるOSS設計はあるが流用はライセンス的に要注意） |
| 保守性 | ○ ブラウザのUI変更に追従する必要がない。ドメインリストの更新のみで対応継続できる |
| 抜け道 | △ DoH（DNS over HTTPS）を使うアプリ・ブラウザはDNSベースの監視を素通りする可能性がある（YouTube/Chromeは条件次第でDoHを使う）。対策にはSNI（TLSのClientHello）レベルの監視が必要になり実装がさらに重くなる |
| 開発速度（MVP到達） | △ VPN+パケット処理の実装・デバッグに時間がかかる |
| 副作用・UX | △ Android標準のVPN常時表示アイコンが出る。他のVPNアプリと同時使用不可（コンフリクト） |
| アプリブロックへの貢献 | ◎ ドメインを潰すことで、ブラウザだけでなく対象アプリ自体の通信も副次的に止まる（フォアグラウンド監視と二重で効く） |

## 案B：まず主要ブラウザ（Chrome等）のみAccessibilityServiceで対応

`AccessibilityService`で対象ブラウザ（例：Chrome）のUIツリーを読み、アドレスバーのURLが
ブロック対象に該当したら検知してブロック動作（戻る/オーバーレイ表示等）を行う。

| 軸 | 評価 |
|---|---|
| ブラウザ非依存の堅牢性 | × 対応していないブラウザ（Firefox、Brave、Samsung Internet、Edge、各種アプリ内WebView等）は素通り。ユーザー自身が抜け道を知っていれば即回避できる |
| 実装コスト | ○ VPN/パケット処理が不要。AccessibilityServiceでUIノードを読むだけなので実装は軽い |
| 保守性 | × ブラウザ側のUI更新でノードID・レイアウトが変わるたびに追従が必要（対象ブラウザが増えるほど保守コストが線形に増える） |
| 抜け道 | × 未対応ブラウザ／WebViewが常に抜け道になる。本人が自分でブロックを設定する用途なので「知っていれば回避できる」状態は要件の趣旨（抑止）に対して弱い |
| 開発速度（MVP到達） | ◎ 対象ブラウザ1つに絞れば最短でMVPが組める |
| 副作用・UX | ○ VPN関連の制約・表示は一切ない |
| アプリブロックへの貢献 | なし（ブラウザ監視の仕組みはアプリブロックには無関係。アプリブロックは別途フォアグラウンド監視で対応——これは案A/Bどちらでも必要な共通部分） |

## 案C：AccessibilityServiceでメジャーブラウザを複数まとめてカバー（2026-06-25追記）

ユーザーから「以前使っていた『AppBlock』はVPNをONにしなくてもどのブラウザでもブロックできていた」
との情報提供あり。VPNを使わずに済んでいる以上、AppBlockは案Aではなく**案Bの発展形**——
1ブラウザ限定ではなく、Chrome/Firefox/Samsung Internet/Edge/Operaなど**主要ブラウザ複数種それぞれに
対応したアドレスバー検出ロジックを最初から複数持っている**AccessibilityServiceベースの実装と推測される
（仕組みとして新しいものではなく、案Bの対応範囲を広げただけ）。

| 軸 | 評価 |
|---|---|
| ブラウザ非依存の堅牢性 | ○ 主要ブラウザを最初から複数（5〜6種程度）カバーすれば、実用上ほとんどのケースに対応できる。マイナーブラウザのみ残存リスク |
| 実装コスト | ○〜△ 案Bと同程度の仕組みを複数ブラウザ分用意するだけ。VPN/パケット処理は不要 |
| 保守性 | △ 対応ブラウザの数だけUI変更追従コストが乗る（案Bより重いが、1ブラウザ限定よりは現実的な落としどころ） |
| 抜け道 | ○ 対応外ブラウザは依然抜け道だが、対応範囲を広げれば実用上のハードルとして十分に機能する |
| 開発速度（MVP到達） | ○ 案Bの延長なので、対応ブラウザを1つずつ追加していく形でMVPからインクリメンタルに拡張できる |
| 副作用・UX | ○ VPN関連の制約なし |
| 既知の運用上の弱点（ユーザー体験談） | **AccessibilityServiceはAndroid側の都合（バッテリー最適化・未使用アプリの権限自動剥奪・メーカー独自の省電力機能等）で、ユーザーの意図に関わらずOFFにされることがある**。これはAppBlockの実装の問題ではなく、AccessibilityServiceという仕組み自体がAndroidに依存する弱点——案B/Cいずれを選んでも同じ制約を受ける。対策としては、サービスがOFFになったことをアプリ側で検知してユーザーに再有効化を促す通知を出す、バッテリー最適化の対象外にするようユーザーに案内する、などの緩和策はあるが「絶対にOFFにならない」保証はできない |

## まとめ・所感

- 「アプリブロック」自体は案A/Bどちらを選んでも別途フォアグラウンド監視で対応するので、
  ここの比較に影響しない（共通でやることが決まっている）。
- 論点になるのは「サイトブロックの抜け道のなさ」と「MVPに到達する速さ」のトレードオフ。
  - 案Aは抜け道が少なく仕組みとして筋が良いが、実装コストが高くMVPが遠のく。DoH対応まで
    含めると更に重くなる。
  - 案Bは最速でMVPに到達できるが、対応ブラウザ以外は素通りする。本人利用の自制目的としては
    「知っていれば回避できる」弱点が、抑止効果そのものを損なうリスクがある。
- 個人開発で「まず動くものを早く出して試す」ことを優先するなら案B/C→後で案Aに拡張、
  「最初から抜け道を作らない」ことを優先するなら案A、という構図。判断軸は実装速度を
  優先するか、抑止力の堅牢性を優先するか。
- ユーザーの実体験（AppBlockで満足できていた）を踏まえると、**案C（主要ブラウザ複数対応の
  AccessibilityService方式）が実装コストと堅牢性のバランスとして現実的な落としどころ**と考えられる。
  ただしAccessibilityServiceがAndroid側都合でOFFにされる既知の弱点があり、これは案B/Cどちらでも
  解消できない——緩和策（再有効化を促す通知等）の実装は必要。
