# アーキテクチャ検討（2026-06-25）

確定済み要件（`01 初回開発/01 要件/めも.txt`参照）を前提に、技術方針をまとめる。
ユーザー確認待ち——実装着手前にこの内容で合意を取ること。

## 全体構成

**サーバーなし、Androidアプリ単体で完結させる。** 前回PJ（将棋盤解析アプリ）はCNN推論のためFastAPIサーバーが必要だったが、本アプリは在宅判定・ブロック判定・通知制御のすべてが端末内で完結する処理であり、外部サーバーを置く理由がない。個人利用・サイドロード前提（配布スコープ確定済み）なので、単一Androidアプリ（Kotlin + Jetpack Compose、前回PJと同じスタック）として実装する。

## 機能ごとの設計

### 1. 在宅判定（Wi-Fi / GPS、おおよそでよい）

| 軸 | Wi-Fi（自宅SSID接続検知） | GPS（ジオフェンス、半径100〜150m） |
|---|---|---|
| バッテリー消費 | ◎ 接続状態の変化イベントを受け取るだけで済み非常に軽い | ○ `GeofencingClient`は生GPSポーリングより省電力だが、Wi-Fiより消費は大きい |
| 屋内での精度 | ◎ 接続できていれば確実（電波が届く範囲＝家とほぼ一致） | △ 屋内（特にコンクリート造）はGPS精度が落ち、20〜100m程度ずれることがある |
| 検知できないケース | × 自宅でWi-Fiをオフにしている／電波が弱くてつながっていない時は「非在宅」と誤判定する | ◎ Wi-Fiの接続状態に関係なく、物理的な位置だけで判定できる |
| 必要な権限 | `ACCESS_FINE_LOCATION`（Android的にSSID取得にも位置情報権限が必要、Wi-Fi自体の権限だけでは済まない） | `ACCESS_FINE_LOCATION` + `ACCESS_BACKGROUND_LOCATION`（バックグラウンドでジオフェンスイベントを受け取るため、より重い権限） |
| 実装コスト | ○ 接続状態の変化を監視するだけで簡単 | △ ジオフェンス登録・座標設定・権限フローがWi-Fiより一手間多い |

**結論・推奨**：精度・実装コストの両面でWi-Fi判定の方が現実的。**まずWi-Fiのみで作り、「Wi-Fiをオフにしている時だけ非在宅と誤判定される」が実用上問題になったらGPSを追加する**、という段階的な進め方を提案します（前回の「案B→必要なら案Aに拡張」と同じMVP優先の考え方）。最初からORで両方実装するのは権限・実装の両方でコストが高くなるため。

- 判定結果は「在宅/非在宅」の状態としてアプリ内に保持し、ルール評価のたびに参照する。

### 2. ルールエンジン（時間帯×在宅判定のAND条件）

- ルールは以下を持つデータ構造として複数定義できる：
  - 時間帯（開始〜終了時刻）
  - 曜日指定（曜日リスト／平日のみ／休日・祝日のみ、複数パターンに対応）
  - 在宅条件（「在宅時のみ」「在宅判定を問わない」のどちらか、ルールごとに選べる）
  - 対象アプリ・サイトのセット（ルールごとに別々に設定可能。例：在宅ルールはYouTube中心、夜間ルールはLINE・メールも含めた広いセット）
- 祝日判定は固定の曜日パターンでは表現できないため、**祝日データは自動更新する**：無料で公開されている祝日API（例：`holidays-jp`系の静的JSON配信）にアプリから定期的にHTTPS GETしてキャッシュする方式を採用する。これは「自社サーバーを持たない」方針とは矛盾しない（読み取り専用の公開データを叩くだけで、自分でサーバーを運用・維持する必要はない）。
  - 起動時・月1回程度のタイミングで取得し、ローカル（DB/ファイル）にキャッシュする。
  - オフラインや取得失敗時は、直前に取得できたキャッシュを使い続ける。さらに保険として、アプリ同梱の当年分の祝日データ（ビルド時点の固定値）をフォールバックとして持っておく（API自体が長期的に止まるリスクへの対策）。
  - 個人開発・無料利用前提のAPIに依存する形になるため、「絶対に途切れない」保証はないが、フォールバックを用意しておけば実用上十分。
- ルール評価は「いずれかのルールに該当する」かどうかで、対象アプリ・サイトの集合を都度計算する（複数ルールが同時に該当する場合は対象を合算）。

### 3. サイト・アプリブロックの実行（案C：AccessibilityService）

- 単一の`AccessibilityService`で以下を行う：
  - フォアグラウンドのウィンドウ変化イベント（`TYPE_WINDOW_STATE_CHANGED`等）でパッケージ名を取得し、現在有効なルールの「ブロック対象アプリ」に該当すれば即座にホーム画面に戻す/オーバーレイ表示する。
  - パッケージがBrave/Chromeの場合、UIツリーからアドレスバーのURL文字列を読み取り、ブロック対象ドメインに該当すれば同様に遮断する。Brave/ChromeはどちらもChromiumベースのため、アドレスバーのリソースID（`url_bar`相当）は共通のものが使える可能性が高いが、実機で個別に確認が必要。
  - **LINEはサイト（URL）単位の制御は行わず、アプリ単位のブロックのみ対応する**（ユーザー確定：2026-06-25）。在宅ルール・時間帯ルールどちらでも、対象に「LINE」を含めれば該当時間帯・在宅状況でアプリそのものを開けなくする。URLフィルタリングのような複雑なロジックは持たない。
- 既知の弱点（要件メモにも記載済み）：AccessibilityServiceはAndroidのバッテリー最適化・未使用アプリの権限自動剥奪等でOSにOFFにされることがある。対策：アプリ起動時・定期チェックでサービスの有効状態を確認し、OFFになっていたら再有効化を促す通知を出す。

### 4. 通知のマナーモード自動切替

- `AudioManager.setRingerMode(RINGER_MODE_VIBRATE)`をルール開始時に呼び、ルール終了時に元のモード（`RINGER_MODE_NORMAL`）に戻す。
- Android 7以降、サイレント以外のリンガーモード変更には`ACCESS_NOTIFICATION_POLICY`権限（設定画面でユーザーが許可）が必要——起動時に権限チェックし、未許可なら設定画面へ誘導する。
- 確定要件どおり、Android標準のDND（おやすみモード）は使わず、リンガーモードの切替のみで対応する。

### 5. バックグラウンド実行の安定性

- ルール評価（時間帯チェック・在宅判定の反映）は`WorkManager`の定期実行、またはAlarmManagerでの時刻トリガーと、Wi-Fi/位置情報の変化イベントの両方をトリガーにする。
- バッテリー最適化の対象外にするようユーザーに案内する（個人利用なので一度設定すれば良い）。
- アプリ全体を1つのフォアグラウンドサービス（常駐通知あり）でまとめておくと、Androidに殺されにくくなる——前回PJのCameraXサービスとは性質が違うが、同様に「常時動くべき処理は明示的にフォアグラウンド化する」方針を踏襲する。

## 必要な権限一覧

| 権限 | 用途 |
|---|---|
| `ACCESS_FINE_LOCATION` / `ACCESS_BACKGROUND_LOCATION` | Wi-Fi SSID取得・ジオフェンスによる在宅判定 |
| `ACCESS_WIFI_STATE` | 現在接続中のWi-Fi SSID取得 |
| `BIND_ACCESSIBILITY_SERVICE`（ユーザーが設定画面でON） | サイト・アプリブロックの実行 |
| `ACCESS_NOTIFICATION_POLICY`（ユーザーが設定画面でON） | マナーモード切替 |
| `SYSTEM_ALERT_WINDOW` | ブロック時のオーバーレイ表示（必要であれば） |
| バッテリー最適化除外（ユーザーが設定画面でON） | バックグラウンド処理の安定性 |

## ユーザー確認結果（2026-06-25）

1. サーバーなしの単一Androidアプリ構成 → **確定**。
2. 在宅判定の方式 → **Wi-Fi判定のみでMVPを作る方針を提案**（精度・実装コストともにWi-Fiが優位、GPSは必要になったら後で追加）。最終確認待ち。
3. LINEの扱い → **アプリ単位のブロックのみで確定**（URL/サイト単位の制御は不要）。
4. 祝日データ → **自動更新（公開APIから取得しローカルキャッシュ＋フォールバック）の方針で確定**（「自動がいい」との回答を受け、実現可能と判断）。
